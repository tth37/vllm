═══════════════════════════ Analysis ═══════════════════════════

1. V1 Anomaly Resolved: Host + MP + TP=2 is NOT slow
─────────────────────────────────────────────────────
   v1 result: Mean TPOT=22.09ms, Median TPOT=5.39ms (4x divergence!)
   v2 result: Mean TPOT=3.22ms,  Median TPOT=3.24ms (healthy convergence)

   Root cause of v1 anomaly:
   - request_rate=inf dumped all 50 requests simultaneously
   - num_warmups=0 caused CUDA graph compilation to hit first requests
   - Identical input/output lengths (128/128) created degenerate batching
   The v2 methodology (ShareGPT + warmup + bounded rate) eliminates all three issues.


2. Configs 1-4 are virtually identical
──────────────────────────────────────
   Throughput: ~991 tok/s (±3 tok/s)
   Median TPOT: 3.24-3.61ms (spread = 0.37ms)
   P99 TTFT: 17-22ms

   Key finding: Docker container overhead is negligible (<0.5% throughput loss).
   TP=2 shows slightly BETTER per-token latency than TP=1 (3.24 vs 3.45ms)
   because the model layers are split across two GPUs.


3. Docker Backend (configs 5-6) shows significant degradation
─────────────────────────────────────────────────────────────
   Throughput:     783-789 tok/s   (21% lower than configs 1-4)
   Median TPOT:   4.20-4.31ms     (25% higher)
   P99 TTFT:      6568-6780ms     (300x worse!)
   Duration:      25.4-25.6s      (vs 20.2s for configs 1-4)

   The throughput drop comes from TWO sources:

   a) Per-token latency overhead (~1ms per step)
      Median TPOT: 4.20ms vs 3.24ms = +0.96ms
      This is the TCP MessageQueue serialization/deserialization overhead.

   b) Scheduling pipeline stall (DOMINANT factor)
      The docker backend disables async scheduling. Without it:
      - Schedule → Execute → Wait → Schedule → Execute → Wait (sequential)
      vs async:
      - Schedule N+1 overlaps with Execute N (pipelined)

      Under sustained 10 req/s load, the sync scheduling can't keep up,
      causing request queueing. Evidence:
      - Mean TTFT: 993-1036ms (vs 14ms) — requests wait in queue
      - P99 TTFT: 6568-6780ms — worst-case queue time
      - Benchmark takes 25.5s (vs 20.2s) for same requests

5. Recommended Optimizations
────────────────────────────
   Priority 1: Enable async scheduling for docker backend
     This would eliminate the 21% throughput drop and 300x P99 TTFT
     degradation. The docker executor's collective_rpc already supports
     non_block=True and FutureWrapper — the main blocker is validation.

   Priority 2: Use SHM MessageQueue when --ipc host is available
     Would reduce the per-token overhead from ~1ms to near-zero.
     Requires setting VLLM_RPC_BASE_PATH on the host executor process.
