═══════════════════════════ Analysis ═══════════════════════════

Why async scheduling doesn't help:

The batch queue (enabled by async scheduling) overlaps SCHEDULING of batch N+1
with GPU EXECUTION of batch N. This saves the scheduling CPU time per step.

For the multiproc executor, this is effective because:
  Step time = GPU_time + SHM_MQ_send(~0.01ms) + Schedule(~0.1ms)
  Overlap savings ≈ Schedule time ≈ 0.1ms per step

For the docker executor, the step time is dominated by TCP overhead:
  Step time = GPU_time + TCP_serialize_send(~0.5ms) + TCP_recv_deserialize(~0.5ms)

The TCP overhead is on the CRITICAL PATH of each step:
  1. Executor serializes + sends via TCP   (~0.5ms)
  2. Worker deserializes                    (~0.1ms)
  3. Worker runs GPU inference              (~3.0ms)
  4. Worker serializes + sends response     (~0.3ms)
  5. Executor receives + deserializes       (~0.1ms)
  Total per step: ~4.0ms  (vs ~3.1ms for multiproc SHM)

The worker processes requests SEQUENTIALLY regardless of how the executor
schedules them. The batch queue just pre-sends the next request, but the
worker can't start it until the current one finishes. The ~0.1ms overlap
from pipelining the schedule() call is negligible compared to the ~1ms
TCP round-trip overhead.

Request pileup analysis:
  Effective step time: ~4.0ms → max throughput ~250 steps/s
  With ShareGPT (avg ~148 output tokens per request), this gives:
    250 * 148 / 136_successful_requests ≈ 272 req/s capacity... not the bottleneck

  Actually the bottleneck is different — with continuous batching and
  max_model_len=512, the engine processes many tokens per step. At 10 req/s,
  the incoming load exceeds the per-step capacity when batches get large.
  The TCP overhead reduces the per-step throughput by ~20%, pushing the
  system closer to saturation and causing queueing.


═══════════════════════ Conclusion ════════════════════════════

The 21% throughput gap between DockerBE and Host/Docker MP is NOT caused by
disabled async scheduling. It is caused by the per-step TCP MessageQueue
overhead (~1ms per step) which is on the critical path and cannot be
overlapped away.

To close this gap, the only viable approach is:
  → Switch from TCP MessageQueue to SHM MessageQueue for co-located containers

This requires:
  1. Set VLLM_RPC_BASE_PATH on the HOST executor process (to the shared volume)
     so ZMQ IPC sockets are accessible from containers
  2. Change n_local_reader from 0 to world_size in the broadcast MQ
  3. Change n_local_reader from 0 to 1 in the response MQ (worker entrypoint)
  4. Rebuild docker image (entrypoint change)

Expected impact: Eliminate ~1ms per step, bringing DockerBE throughput
to within ~3% of Host MP (matching Docker Container MP performance).
