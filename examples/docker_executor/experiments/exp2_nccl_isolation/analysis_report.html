1. Objective

Quantify the NCCL communication bandwidth loss when GPU workers run in per-GPU isolated containers (one physical GPU per container, private IPC and network namespaces) on NVLink-connected hardware. The experiment tests whether environment-variable-only configurations can recover NVLink P2P performance without code changes to NCCL.

2. Background

NCCL selects transport based on what the runtime environment makes available. On NVLink-connected GPUs, the optimal path is P2P/CUMEM/read, which performs direct GPU-to-GPU DMA over NVLink via CUDA IPC. Three barriers prevent this under container isolation:

  1. Hostname mismatch — containers have different hostnames, causing NCCL to treat them as separate nodes. Solved with NCCL_HOSTID.
  2. GPU topology invisible — per-GPU CUDA_VISIBLE_DEVICES hides the interconnect topology. Partially solved with NVIDIA_VISIBLE_DEVICES=all (NVML can see topology).
  3. CUDA peer access impossible — each container sees only one GPU as cuda:0. cudaDeviceCanAccessPeer() fails because there is no peer device. This cannot be solved with environment variables.

6. Analysis

6.1 Environment variables alone cannot restore NVLink P2P

The fundamental barrier is cudaDeviceCanAccessPeer(): with per-GPU CUDA_VISIBLE_DEVICES, each container sees only cuda:0. There is no peer device to check, so NCCL’s P2P transport selection (p2pCanConnect) fails unconditionally. No combination of NCCL_HOSTID, NVIDIA_VISIBLE_DEVICES, or other environment variables can change this — it is a CUDA runtime constraint.

6.2 SHM is the best achievable fallback without code changes

By sharing /dev/shm and /tmp as Docker volumes and setting NCCL_P2P_DISABLE=1 + NCCL_HOSTID, NCCL selects SHM/direct/direct transport. This routes data through host DRAM (CPU memory), achieving 13–23 GB/s — about 9–12% of the NVLink P2P baseline (157–261 GB/s). While substantially better than the naive TCP fallback (4–9 GB/s), it still represents a ~7–12× degradation.

6.3 Without SHM workaround, NCCL falls to TCP sockets

The naive isolation config (no shared /dev/shm, no NCCL_P2P_DISABLE) causes NCCL to detect nNodes=2 and fall back to NET/Socket transport over the Docker bridge network. This achieves only 2–5% of baseline bandwidth. The SHM workaround provides a 3–4× improvement over this TCP baseline.

6.4 The penalty is architectural, not configurational

NVLink P2P transport requires CUDA IPC (cudaIpcGetMemHandle / cudaIpcOpenMemHandle), which requires both GPUs to be visible in the same CUDA context. Per-GPU container isolation makes this impossible by design. Recovering NVLink performance would require one of:

  1. Patching NCCL to use pidfd_getfd() (kernel 5.6+) for cross-namespace CUDA IPC file descriptor passing
  2. Using the CUDA VMM API (cuMemExportToShareableHandle with CU_MEM_HANDLE_TYPE_POSIX_FILE_DESCRIPTOR) to export GPU memory as POSIX FDs shareable via Unix domain sockets
  3. Giving each container visibility to all GPUs (defeats isolation)

7. Conclusions

  1. Per-GPU container isolation on NVLink hardware causes an 88–91% NCCL bandwidth loss compared to the P2P/NVLink baseline. This is a fundamental CUDA runtime constraint, not a configuration problem.
  2. The best environment-variable-only workaround (shared /dev/shm + NCCL SHM transport) recovers to ~10% of baseline, achieving 13–23 GB/s through host DRAM.
  3. Without the SHM workaround, NCCL falls back to TCP sockets at 2–5% of baseline (4–9 GB/s).
  4. Restoring NVLink P2P performance under per-GPU isolation requires NCCL source-level modifications to enable cross-container CUDA IPC.
