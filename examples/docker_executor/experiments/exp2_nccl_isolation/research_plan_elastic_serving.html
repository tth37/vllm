Research Plan: Elastic GPU Inference Serving
via Container-Isolated Tensor Parallelism

Direction B · Building on Exp1 + Exp2 Results
Draft 2026-04-03 (related work updated)

1. Motivation

Experiments 1 and 2 established the communication foundation: vLLM’s DockerDistributedExecutor can run per-GPU worker containers with full NVLink bandwidth recovery via NCCL’s cuMem VMM API. The remaining ~10% TPOT gap is due to vLLM’s custom allreduce kernel using cudaIpcMemHandle (an engineering fix, not a research problem).

With zero-overhead container isolation proven, the question becomes: what can we do with individually manageable GPU containers that we cannot do with a monolithic multi-GPU process?

Thesis: Per-GPU container isolation enables elastic tensor-parallel inference — live TP resizing, fault recovery, and GPU time-sharing — capabilities that are fundamentally impossible with the conventional multiprocess executor model where all GPUs share a single process lifetime.

1.1 What Container Isolation Enables

CapabilityMonolithic MPContainer BackendWhy It Matters
Independent GPU lifecycle All GPUs share one process; any change requires full restart Add/remove individual GPU containers at runtime Foundation for all elastic operations
Fault isolation One GPU crash kills the entire serving process Crashed container can be restarted independently Availability SLA for production deployments
Live TP resizing Impossible — TP degree is fixed at startup Scale TP=2 → TP=4 by adding containers + repartitioning Adapt to load without service interruption
GPU time-sharing GPU is locked to one model for the process lifetime Swap containers on/off a GPU for multi-tenant serving GPU utilization in multi-model deployments
Heterogeneous mixing All GPUs must be identical (same CUDA context) Different GPU types can join the same TP group (if sharding supports it) Utilize mixed GPU fleets

1.2 Why This Is a Research Problem

None of the above capabilities are straightforward to implement. The key challenges are:

  1. NCCL communicator rebuild: NCCL communicators are created once at initialization with a fixed set of ranks. Adding or removing a rank requires ncclCommAbort() on all participants and a full ncclCommInitRank() with the new world size. This is a collective operation — all surviving ranks must participate, and the new communicator must be established before inference can resume.
  2. Model shard redistribution: When TP degree changes (e.g., TP=2 → TP=4), model weights must be re-sharded and redistributed. This requires either (a) re-loading from disk/checkpoint on each new shard, or (b) live weight transfer from existing shards to new ones via NCCL or CUDA IPC.
  3. In-flight request draining: During TP resize or fault recovery, in-flight requests in the scheduler’s continuous batching loop must be handled. Options: drain and re-enqueue, speculative continuation on surviving ranks, or request-level checkpointing.
  4. KV cache invalidation: TP degree change invalidates the entire KV cache (different head partitioning). All in-progress sequences must be re-prefilled on the new TP group. Minimizing this cost is critical for live resize latency.

2.2 GPU Container Isolation and NCCL Communication

The central technical problem of this work is achieving high-bandwidth GPU-GPU communication in per-GPU-isolated containers. The industry-wide status quo can be categorized into three camps:

Camp 1: Sacrifice Isolation (Industry Standard)

The dominant approach gives all GPUs to a single container (or uses --ipc=host --network=host with all GPUs visible), providing full NCCL bandwidth but no isolation:

  • Kubernetes nvidia-device-plugin: Assigns GPUs to pods but relies on hostIPC: true and all GPUs visible for multi-GPU NCCL jobs. The device plugin has no mechanism to expose topology without exposing all GPUs.
  • Cloud providers (AWS EKS, GCP GKE, Azure AKS): All use the all-GPUs-in-one-container model for distributed training/serving on p4d/p5, A3, NDv4/NDv5 instances.
  • PyTorch Elastic (torchrun): Supports elastic membership at the node level (add/remove machines), but within a node all GPUs share one process group. No per-GPU container isolation.
  • Ray + vLLM: Ray workers are OS processes, not containers. GPU isolation is via CUDA_VISIBLE_DEVICES but processes share IPC/network namespace. NCCL sees all GPUs via shared /dev/nvidia*.

Camp 2: Accept Performance Loss

Per-GPU container isolation is achievable but forces NCCL to fall back to TCP socket transport:

  • Our exp1 single_visible_gpu variant: Each container sees one GPU with private IPC. NCCL detects nNodes=N, localRanks=1 and uses NET/Socket/2 transport. Result: 2–10× TPOT degradation (e.g., 245 ms vs 26 ms for Qwen3-14B TP=4).
  • NVIDIA MIG (Multi-Instance GPU): Hardware partitioning that explicitly disables P2P between GPU instances. NCCL across MIG slices uses host-staged SHM at best, TCP at worst.
  • Singularity/Apptainer (HPC): Uses bind-mount isolation rather than Linux namespaces. GPUs are visible via host device files, so NCCL works, but this provides no security isolation.

Camp 3: Emerging Official Support (Not Yet Deployed)

  • NVIDIA IMEX Daemon (CUDA 12.4+): A kernel-level mechanism providing /dev/nvidia-caps-imex-channels for cross-container GPU memory exchange. This is architecturally complementary to our cuMem approach — NVIDIA is building official infrastructure for the same problem space. However, it requires a specific kernel module, is not yet widely deployed, and its documentation does not address the per-GPU container isolation use case explicitly. The IMEX approach works at the driver level, while our cuMem VMM + UDS approach works at the NCCL transport level.
  • NVIDIA DRA (Dynamic Resource Allocation) plugin for Kubernetes: A next-generation device plugin using Kubernetes’ DRA framework. Supports finer-grained GPU resource claims but does not address NCCL transport selection or topology exposure.

2.3 NCCL Internals: Three Gates for Intra-Node Transport

Through source-level analysis of NCCL (v2.21+), we identified three independent checks that all must pass for NCCL to select P2P or SHM transport instead of falling back to TCP:

GateWhat NCCL ChecksWhat Breaks ItFix
1. hostHash hash(hostname + /proc/sys/kernel/random/boot_id) — computed in getHostHash() (src/misc/utils.cc) Different container hostnames produce different hashes, making NCCL treat containers as separate nodes NCCL_HOSTID=<same_value> replaces the entire hash input string
2. shmDev stat("/dev/shm").st_dev — the device ID of the tmpfs mount backing /dev/shm Each container gets its own tmpfs with a unique device ID Mount a shared /dev/shm volume (--volume /dev/shm:/dev/shm)
3. IPC socket NCCL uses abstract Unix domain sockets for P2P/SHM setup handshakes; abstract UDS are scoped to the network namespace Separate network namespaces prevent cross-container UDS connections --network=host, or NCCL_IPC_USE_ABSTRACT_SOCKET=0 + bind-mount a shared filesystem socket directory

These three gates are undocumented as a set in NCCL’s official documentation. Our contribution is identifying them through source analysis and demonstrating that satisfying all three, combined with NCCL_CUMEM_ENABLE=1, recovers full NVLink bandwidth in per-GPU containers.

2.4 NCCL cuMem VMM Transport Path

The cuMem VMM API (CUDA 12.2+) provides a fundamentally different IPC mechanism from legacy cudaIpcMemHandle:

AspectLegacy cudaIPCcuMem VMM
Handle type Opaque 64-byte struct, shared via IPC namespace POSIX file descriptor, shared via sendmsg() + SCM_RIGHTS
Namespace requirement Shared IPC namespace (--ipc=host) Shared network namespace (for abstract UDS) or filesystem UDS
Peer GPU visibility Both GPUs must be in CUDA_VISIBLE_DEVICES Peer GPU can be invisible; driver resolves the mapping via bus ID
NCCL code path P2P_IPC subtype in transport/p2p.cc P2P_CUMEM subtype; enabled by NCCL_CUMEM_ENABLE=1
Container isolation Incompatible with per-GPU isolation (requires shared IPC + all GPUs visible) Compatible: only needs shared network namespace + topology exposure

The cuMem path is the key mechanism that enables our approach. With NCCL_CUMEM_ENABLE=1, NCCL’s p2pCanConnect() proceeds even when busIdToCudaDev() returns −1 for the peer GPU (i.e., the peer is not in CUDA_VISIBLE_DEVICES). The driver resolves the physical NVLink/PCIe path internally.

2.7 Positioning Summary

Gap: No existing system, framework, or published technique achieves GPU-granularity elastic tensor parallelism with per-GPU container isolation AND high-bandwidth interconnect. The entire ecosystem either (a) sacrifices isolation for performance, (b) accepts catastrophic bandwidth loss (2–10×) under isolation, or (c) operates at instance/node granularity rather than GPU granularity.

Our contribution is two-fold:

  • Mechanism: Identifying the three NCCL gates (hostHash, shmDev, IPC socket) and demonstrating that NCCL_CUMEM_ENABLE=1 combined with proper container configuration recovers full NVLink/PCIe bandwidth under per-GPU isolation — a configuration that is undocumented in NCCL, NVIDIA, or Kubernetes documentation.
  • Application: Using this mechanism to enable fault recovery, live TP resizing, and elastic multi-model GPU scheduling — capabilities that are impossible with the conventional monolithic multi-GPU process model.

3. Proposed Experiments

Three experiments, each building on the previous, culminating in a prototype elastic serving system.

Exp3: Single-GPU Fault Recovery

Goal: Kill a GPU worker container mid-inference, restart it, rejoin the NCCL communicator, reload its model shard, and resume serving. Measure recovery time and request impact.

3.1 Design

ComponentMechanismKey Challenge
Failure detection Host executor detects container exit or MQ timeout Distinguishing transient MQ delay from true failure
Communicator teardown ncclCommAbort() on all surviving ranks Must be called on every rank; blocking collective
Container restart Host executor launches replacement container with same rank/GPU New container must join NCCL init with existing ranks
Communicator rebuild ncclCommInitRank() with same world_size, new unique ID All ranks must synchronize; new container needs the ncclUniqueId
Model reload New container loads its shard from HF cache (already mounted) Model loading time dominates recovery latency
Request handling Scheduler pauses dispatch; in-flight requests drain or re-enqueue KV cache for in-flight sequences is lost on the failed rank

3.2 Metrics

  • Recovery latency: Wall time from container death to first successful inference on the rebuilt TP group (target: <30s for 4B model, <60s for 14B)
  • Request impact: Number of requests that must be re-prefilled (all in-flight sequences lose their KV cache on the failed rank)
  • Throughput dip: Duration and depth of the throughput drop during recovery
  • Correctness: Output quality is identical to a fresh start (bitwise NCCL reproducibility is not guaranteed, but statistical equivalence)

3.3 Prototype Implementation Sketch

# In docker_executor.py — new recovery path

class DockerDistributedExecutor:

    def _detect_worker_failure(self, rank: int) -> bool:
        """Check if worker container is still alive."""
        result = subprocess.run(
            ["docker", "inspect", "-f", "{{.State.Running}}",
             self.container_ids[rank]],
            capture_output=True, text=True)
        return result.stdout.strip() != "true"

    def _recover_failed_worker(self, failed_rank: int):
        """Rebuild the TP group after a single-GPU failure."""
        # 1. Signal all surviving workers to abort NCCL communicator
        self._broadcast_abort_signal()

        # 2. Wait for surviving workers to confirm abort
        self._wait_for_abort_ack(timeout=10)

        # 3. Generate new ncclUniqueId
        new_nccl_id = self._generate_nccl_unique_id()

        # 4. Launch replacement container for failed_rank
        self._launch_worker_container(
            rank=failed_rank,
            nccl_unique_id=new_nccl_id)

        # 5. Signal surviving workers to reinitialize with new ID
        self._broadcast_reinit_signal(new_nccl_id)

        # 6. Wait for all workers (including replacement) to complete
        #    NCCL init + model load
        self._wait_for_ready(timeout=120)

        # 7. Resume scheduler — all in-flight requests re-prefilled
        self._scheduler_resume()

3.4 Test Protocol

  1. Start vLLM with DockerBE + CUMEM isolation, TP=2 (or TP=4), Qwen3-4B
  2. Run sustained load (10 req/s ShareGPT) for 60s to reach steady state
  3. docker kill <worker_rank1_container> at t=60s
  4. Measure time until serving resumes (first successful completion after kill)
  5. Compare output quality: same prompts before and after recovery
  6. Repeat 5 times for statistical significance

Open question: Should surviving workers continue serving at reduced TP during recovery? For TP=4 → TP=3, this would require dynamic TP support (non-power-of-2 TP is not standard). Alternative: pause all serving during recovery (simpler, but higher availability impact).

Exp4: Live TP Resizing

Goal: Scale from TP=2 to TP=4 (and back) without stopping the serving process. The model is re-sharded live, new GPU containers join, and inference continues with minimal interruption.

4.1 Design

Live TP resize is strictly harder than fault recovery because it requires model weight redistribution in addition to communicator rebuild. Two approaches:

ApproachMechanismProsCons
A: Reload from disk All containers (old + new) reload their new shard from the HF cache with updated TP config Simple; reuses existing model loading code; no cross-container data transfer Slow (~30s for 14B model); wastes I/O bandwidth; requires all ranks to reload
B: Live shard transfer Old TP group sends weight slices to new containers via NCCL or CUDA IPC; new containers only receive their shard Fast (GPU-to-GPU over NVLink); existing shards don’t need to reload Complex; requires custom weight redistribution logic; two communicator groups coexist briefly

Recommendation: Start with Approach A (reload from disk) for the prototype. It is simpler and establishes the control-plane machinery (communicator rebuild, scheduler pause/resume, KV cache invalidation). Approach B is an optimization for the final system.

4.2 Scale-Up Sequence (TP=2 → TP=4)

  1. Trigger: Orchestrator detects sustained high load or receives explicit scale-up command
  2. Drain: Scheduler stops accepting new requests; in-flight batch completes
  3. Abort: All existing workers call ncclCommAbort()
  4. Launch: Executor starts 2 new GPU containers (ranks 2, 3)
  5. Reconfigure: All 4 containers receive new VllmConfig with tensor_parallel_size=4
  6. Init: All 4 containers call ncclCommInitRank(world_size=4)
  7. Reload: Each container loads its 1/4 shard (vs previous 1/2 shard)
  8. Resume: Scheduler resumes with new TP=4 group; KV cache starts empty

4.3 Scale-Down Sequence (TP=4 → TP=2)

  1. Trigger: Low load detected or explicit scale-down command
  2. Drain: In-flight batch completes
  3. Abort: All 4 workers call ncclCommAbort()
  4. Stop: Executor terminates containers for ranks 2, 3
  5. Reconfigure: Remaining 2 containers receive new VllmConfig with tensor_parallel_size=2
  6. Init: 2 containers call ncclCommInitRank(world_size=2)
  7. Reload: Each container loads its 1/2 shard
  8. Resume: Scheduler resumes with TP=2; freed GPUs available for other models

4.4 Metrics

  • Resize latency: Wall time from trigger to first inference on new TP group
  • Downtime: Duration of zero-throughput window during resize
  • KV cache re-prefill cost: Time to re-prefill in-flight sequences after resize
  • Throughput trajectory: Time series of req/s and tok/s through the resize event
  • Comparison: Resize latency vs. full cold-start of new TP configuration

Open question: Can we preserve KV cache across TP resize? For TP=2 → TP=4, each original rank holds attention heads [0..H/2). After resize, rank 0 holds [0..H/4) and rank 2 holds [H/4..H/2). In principle, rank 0 could send the second half of its KV cache to rank 2 via NCCL. This would eliminate re-prefill cost but adds significant implementation complexity. Worth investigating in a later phase.

5. Technical Deep-Dives

5.1 NCCL Communicator Lifecycle

NCCL communicators are heavyweight objects. Understanding their lifecycle is critical for Phase 1:

# Normal lifecycle (current vLLM):
ncclGetUniqueId(&id)           # Rank 0 generates, broadcasts to others
ncclCommInitRank(&comm, nranks, id, rank)  # All ranks call collectively
# ... use comm for collectives ...
ncclCommDestroy(comm)          # Clean shutdown

# Recovery lifecycle (our extension):
ncclCommAbort(comm)            # Emergency teardown — non-blocking, no sync
                               # Frees resources even if peers are dead
# ... restart failed container ...
ncclGetUniqueId(&new_id)       # Generate fresh ID
ncclCommInitRank(&new_comm, nranks, new_id, rank)  # All ranks (including replacement)
# ... resume serving ...

Key constraint: ncclCommAbort() must be called on every surviving rank. The failed rank’s communicator is implicitly abandoned. After abort, the surviving ranks’ NCCL state is fully released — they can ncclCommInitRank() again.

5.2 Scheduler Pause/Resume Protocol

During recovery or resize, the vLLM scheduler must coordinate with the executor:

  1. Pause: Executor sets a flag; scheduler stops dispatching new batches
  2. Drain: Currently executing batch completes (or is aborted if the failed rank was involved)
  3. Recovery: Executor performs NCCL rebuild + model reload
  4. KV invalidation: All sequences in the scheduler’s running queue are moved back to waiting, marked for re-prefill
  5. Resume: Executor clears the flag; scheduler resumes dispatching

The scheduler already has a running → waiting transition path for preemption (recompute mode). The same mechanism can be reused for post-recovery re-prefill.

5.3 Container Launch Optimization

Recovery latency is dominated by model loading, not container startup. Potential optimizations:

OptimizationExpected SpeedupComplexity
Warm container pool: Pre-launch idle containers that only need NCCL init + model load ~2s saved (container startup) Low
Model shard caching: Keep model weights in a tmpfs/ramdisk; mmap instead of loading from SSD ~5-10s saved (I/O bound → memory bound) Low
Live weight transfer: Surviving ranks send the failed rank’s shard via NCCL (Approach B) ~10-20s saved for large models (NVLink >> SSD) High
Checkpoint-free resume: Store model weights in GPU memory with cuMem; new container maps existing allocation Near-instant shard recovery High (requires cuMem persistence across container restarts)
